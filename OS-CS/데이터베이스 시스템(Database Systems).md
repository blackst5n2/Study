수많은 데이터를 **체계적으로 관리하고 빠르게 검색** 하는 방법.

데이터베이스는 데이터를 '검색하고 관계 짓는' 방법.

# 데이터베이스의 핵심: SQL과 선언적 언어
**데이터베이스 관리 시스템(DBMS)** 은 데이터의 안전성, 동시성, 무결성(Integrity)을 책임지는 복잡한 소프트웨어임. 이 DBMS와 대화하는 언어가 바로 **SQL (Structured Query Language)** 임.

#### 선언적 언어 (Declarative Language)
- 일반적인 프로그래밍 언어(C, Python)는 "어떻게(How)" 해야 하는지를 명시함. (예: "배열을 돌면서 100보다 큰 값을 찾아라.")
- SQL은 "무엇을(What)" 원하는지만 명시함. (예: `SELECT * FROM table WHERE value > 100;`)

DB의 역할: DBMS 내부의 **쿼리 옵티마이저(Query Optimizer)** 가 "이것을 가장 빠르게 수행하는 방법(최적의 실행 계획)"을 스스로 결정하여 실행함. 이는 DBMS 내부 구조에 대한 깊은 이해가 필요.

# 성능의 비밀: B-Tree 인덱싱
DBMS에서 수백만 건의 데이터 중 특정 한 건을 찾는 것이 파일 시스템에서 전체 파일을 순차적으로 읽는 것 (O(N))보다 훨씬 빠른 이유는 **인덱스(Index)** 덕분이며, 대부분의 DBMS는 **B-Tree** 구조를 사용함.

#### 인덱스의 필요성
- 테이블 전체를 처음부터 끝까지 읽는 것을 Full Table Scan이라고 하며, 데이터가 많으면 수십 분이 걸림.
- 인덱스는 책의 **'찾아보기'** 와 같음. 특정 키를 찾기 위해 전체를 읽을 필요 없이, 인덱스만 타고 내려가면 됨.

#### B-Tree 구조의 특징
B-Tree는 **하드디스크(HDD/SDD)의 특성에 최적화된 트리 구조** 임.
1. 저장 단위: B-Tree의 각 노드(Node)는 **디스크의 블록 크기(4KB 또는 그 이상)** 와 거의 일치하도록 설계됨.
2. 낮은 깊이: 각 노드에 **수많은 자식 포인터** 를 가질 수 있음 (Fan-out이 큼). 덕분에 수백만 건의 데이터를 찾아도 **트리의 깊이(Height)** 가 3~5층 정도로 매우 낮음.
3. 검색 속도: 검색에 필요한 디스크 접근 횟수(I/O 횟수)가 극히 적어, 검색 속도가 $O(\log N)$으로 매우 빠름.

	Inode와 연결: 파일 시스템의 Inode는 "파일을 구성하는 블록 목록"을 빠르게 찾게 해주는 **데이터 블록의 인덱스** 역할을 했음. B-Tree는 **"데이터를 구성하는 행(Row)"** 을 빠르게 찾게 해주는 역할을 함.

# 트랜잭션과 ACID 속성 (데이터 무결성)
DBMS의 가장 중요한 임무는 **데이터의 무결성** 과 **동시성 제어** 임. 이는 **트랜잭션(Transaction)** 개념으로 구현됨.
- 트랜잭션: 데이터베이스의 상태를 변화시키는 **하나의 논리적 작업 단위**. (예: 은행 계좌 이체)

트랜잭션이 가져야 하는 네 가지 핵심 속성을 ACID라고 함.

| 속성  | 영어 명칭       | 설명                                                          |
| --- | ----------- | ----------------------------------------------------------- |
| 원자성 | Atomicity   | 트랜잭션은 전부 성공하거나 전부 실패해야 함. (이체 중 오류가 나면 송금/입금이 모두 취소되어야 함.)  |
| 일관성 | Consistency | 트랜잭션은 DB의 상태를 유효한 상태로만 변경해야 함. (잔고가 마이너스가 되는 등의 불가능한 상태 방지) |
| 고립성 | Isolation   | 여러 트랜잭션이 동시에 실행되어도, 각 트랜잭션은 마치 혼자 실행되는 것처럼 보여야 함. (동시성 제어)  |
| 지속성 | Durability  | 트랜잭션이 성공적으로 완료되면, 그 결과는 시스템 오류나 정전에도 불구하고 영구적으로 보존되어야 함.    |

# 임베디드 데이터베이스 (SQLite)
임베디드 분야에서는 서버용 DB(MySQL, PostgreSQL) 대신 SQLite가 널리 사용됨.
- 특징: 별도의 DB 서버 프로세스가 필요 없이, 프로그램 내에 라이브러리 형태로 내장되어 동작함.
- 활용: 안드로이드, iOS, 웹 브라우저, 임베디드 장치 등 단일 애플리케이션이 데이터를 관리할 때 최적화되어 있음.
